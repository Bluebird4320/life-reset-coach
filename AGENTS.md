# アプリ制作ルール / AI開発エージェント指示書

このファイルは、本アプリを **Codex / Claude Code / ChatGPT / 人間の開発者** が安全かつ継続的に開発するための共通ルールです。  
迷った場合は、この順番で優先します。

1. ユーザーが長く使いたくなる体験
2. 誰でも迷わず操作できるUI
3. 壊れにくく、後から直しやすい設計
4. セキュリティ・プライバシー
5. 実装速度

---

## 1. プロジェクトの目的

このアプリは、ユーザーが「叶えたい将来」に向かって日々の行動を整えるための支援アプリである。

特に以下を重視する。

- 毎日開きたくなる、やさしく楽しいUI
- 初心者でも直感的に使える操作性
- 目標・行動・振り返りをシンプルに継続できる仕組み
- 目標に近づいている時は、可愛らしいドット絵の天使でポジティブに演出する
- 目標に反する行動や停滞がある時は、可愛らしいドット絵の悪魔でやさしく気づきを促す
- AIは説教役ではなく、行動を前に進める伴走者として使う

## 2. 絶対に守る非交渉ルール

### 2.1 ユーザー体験優先

- 機能追加よりも、毎日使える分かりやすさを優先する。
- 1画面に情報を詰め込みすぎない。
- 主要操作は原則3タップ以内で完了させる。
- 初回利用時に「何をすればいいか」が即座に分かる状態にする。
- 空状態、読み込み中、エラー時の表示を必ず用意する。

### 2.2 小さく作って壊さない

- 1回の変更で複数の大規模改修をしない。
- 既存機能を壊す可能性がある変更は、必ず影響範囲を確認する。
- 仕様が曖昧な場合は、最も安全で後から変更しやすい実装を選ぶ。
- 不要な抽象化、過剰設計、将来使うか不明な機能を先に作らない。

### 2.3 セキュリティ最優先

- APIキー、トークン、秘密情報をコードに直書きしない。
- `.env` や秘密情報をGitにコミットしない。
- ユーザーの目標、行動履歴、振り返りは個人情報に近いデータとして扱う。
- 外部AI/APIに送る情報は必要最小限にする。
- ログに個人情報・APIレスポンス全文・秘密情報を出さない。

### 2.4 Gitを汚さない

- コミット前に差分を確認する。
- 関係ないファイルを同じコミットに混ぜない。
- 自動生成ファイル、ビルド成果物、キャッシュ、環境依存ファイルをコミットしない。
- コミットメッセージは「何を・なぜ」変更したか分かる内容にする。

## 3. 技術方針

### 3.1 基本スタック

- フロントエンドは Expo / React Native を前提とする。
- 言語は TypeScript を基本とする。
- 開発確認は Expo Go を活用する。
- まずはスマホで自然に使える体験を優先する。
- Web対応は必要になってから追加する。

### 3.2 ライブラリ選定

- ライブラリ追加は慎重に行う。
- 追加前に以下を確認する。
  - 本当に必要か
  - Expo環境で問題なく動くか
  - メンテナンスされているか
  - 代替手段より複雑にならないか
- 小さな処理のために重い依存関係を追加しない。

### 3.3 状態管理

- 最初はReact標準の `useState` / `useReducer` / Context を優先する。
- グローバル状態は必要最小限にする。
- 状態の置き場所は以下の基準で決める。
  - 画面内だけで使う: local state
  - 複数画面で使う: Contextまたは軽量store
  - 永続化が必要: storage層に切り出す

### 3.4 データ保存

- 初期段階ではローカル保存を優先する。
- 将来的なクラウド同期を想定し、データ構造は拡張しやすく設計する。
- データには必ず `id`, `createdAt`, `updatedAt` を持たせる。
- 削除はすぐ物理削除にせず、必要に応じて復元可能な設計を検討する。

## 4. ディレクトリ・ファイル構成ルール

推奨構成は以下。

```txt
src/
  app/                  # 画面・ルーティング
  components/           # 再利用UI部品
  features/             # 機能単位の実装
  hooks/                # カスタムフック
  lib/                  # 汎用関数・外部サービス接続
  stores/               # 状態管理
  types/                # 型定義
  constants/            # 定数
  assets/               # 画像・アイコン・フォント等
  tests/                # テスト
```

### 4.1 ファイル作成ルール

- 1ファイルに責務を詰め込みすぎない。
- UI、ロジック、データアクセスは可能な限り分離する。
- 画面コンポーネントに複雑なビジネスロジックを書かない。
- 型定義は再利用される場合 `types/` に切り出す。
- マジックナンバーや固定文言は `constants/` に切り出す。

### 4.2 命名ルール

- コンポーネント: `PascalCase`
- 関数・変数: `camelCase`
- 定数: `UPPER_SNAKE_CASE`
- ファイル名は役割が分かる名前にする。
- 曖昧な名前を避ける。
  - 悪い例: `data`, `util`, `temp`, `test2`
  - 良い例: `goalProgressCalculator`, `DailyCheckInCard`

## 5. UI/UXルール

### 5.1 基本思想

このアプリのUIは「継続したくなる軽さ」と「未来に進んでいる感覚」を両立する。

- かわいさ、分かりやすさ、行動しやすさを優先する。
- 文字量を減らし、視覚的に状態が分かるようにする。
- ユーザーを責める表現を使わない。
- 悪魔表示は罰ではなく、気づきのサインとして扱う。
- 天使表示は達成感と継続意欲を高める演出として扱う。

### 5.2 ホーム画面ルール

ホーム画面では、ユーザーが一目で以下を理解できること。

- 今日やること
- 現在の方向性が良いか悪いか
- 目標に対する進捗
- 次に取るべき小さな行動

ホーム画面に置く情報は必要最小限にする。詳細情報は別画面に逃がす。

### 5.3 キャラクター演出ルール

- 天使・悪魔は可愛らしいドット絵を基本とする。
- 表示条件はロジックとして明確に定義する。
- 悪魔表示時も、否定的・攻撃的な言葉は使わない。
- 状態変化には小さなアニメーションを付けるとよい。
- キャラクターはアプリの継続率を高めるための要素であり、機能の邪魔をしない。

### 5.4 アクセシビリティ

- 文字サイズは小さくしすぎない。
- 色だけで状態を判断させない。
- ボタンやタップ領域は十分な大きさを確保する。
- コントラスト不足を避ける。
- スクリーンリーダー対応を意識したラベルを付ける。

## 6. AI機能ルール

### 6.1 AIの役割

AIはユーザーの目標達成を支援する補助機能であり、アプリの本体ではない。

AIに任せること。

- 振り返り内容の整理
- 行動の改善提案
- 目標とのズレの言語化
- 次の一歩の提案
- 継続を促す短いメッセージ生成

AIに任せすぎないこと。

- ユーザーの価値観の決定
- 重要な人生判断の断定
- 医療・法律・金融など高リスク領域の断定的助言
- 事実確認が必要な最新情報の無根拠な回答

### 6.2 AI出力の品質ルール

AIの返答は以下を満たすこと。

- 短く、行動に移しやすい
- 責めない
- 抽象論だけで終わらない
- 「今日できる1アクション」を含める
- ユーザーの入力内容に基づく
- 不確実なことは断定しない

### 6.3 プロンプト管理

- プロンプトはコード内に散らばらせない。
- AI用プロンプトは `prompts/` または `lib/ai/` に集約する。
- プロンプトにはバージョンを付ける。
- AI出力のJSON形式を使う場合、必ずバリデーションする。
- JSONパース失敗時のフォールバック表示を用意する。

### 6.4 AI安全設計

- AIが失敗してもアプリの基本機能は使えるようにする。
- AI通信中はローディング表示を出す。
- APIエラー時は、ユーザーに分かりやすく再試行導線を出す。
- ユーザー入力をそのまま外部送信する場合は、将来的に同意UIを設ける。

## 7. 品質管理ルール

### 7.1 実装前チェック

実装前に必ず確認する。

- 何のための変更か
- どの画面・機能に影響するか
- 既存データとの互換性はあるか
- UIが複雑にならないか
- テストすべきケースは何か

### 7.2 実装後チェック

実装後に必ず確認する。

- アプリが起動する
- 対象画面が表示される
- 主要操作が動く
- エラー表示が破綻しない
- TypeScriptエラーがない
- Lint/Formatが通る
- 不要な `console.log` が残っていない
- 差分に秘密情報が含まれていない

### 7.3 テスト方針

優先順位は以下。

1. 目標・行動・進捗計算など壊れると困るロジック
2. AIレスポンスの整形・バリデーション
3. 保存・復元処理
4. 画面の主要操作

最低限、重要な純粋関数にはユニットテストを追加する。

## 8. パフォーマンスルール

- 初回表示を重くしない。
- ホーム画面で不要なAPI通信をしない。
- 画像・ドット絵素材はサイズを最適化する。
- 無限再レンダリングを避ける。
- バッテリー消費の大きい処理を避ける。
- アニメーションは軽量にする。
- 大量データを扱う場合はページングまたは絞り込みを使う。

---

## 9. エラー処理ルール

エラーは開発者向けとユーザー向けを分ける。

### 9.1 ユーザー向け

- 難しい技術用語を出さない。
- 何が起きたかを短く伝える。
- 次にできる行動を提示する。

例。

```txt
うまく読み込めませんでした。
通信状態を確認して、もう一度お試しください。
```

### 9.2 開発者向け

- 原因調査に必要な情報だけログに残す。
- 個人情報や秘密情報はログに出さない。
- エラー原因を握りつぶさない。

## 10. Git / GitHub運用ルール

### 10.1 ブランチ

- `main` は常に動く状態に保つ。
- 作業は原則ブランチを切る。
- ブランチ名は目的が分かる名前にする。

例。

```txt
feature/home-character-status
feature/daily-check-in
fix/progress-calculation
chore/add-app-rules
```

### 10.2 コミット

コミットメッセージは以下の形式を推奨する。

```txt
<type>: <変更内容>
```

例。

```txt
feat: ホーム画面に天使・悪魔ステータスを追加
fix: 進捗計算の判定条件を修正
chore: アプリ制作ルールを追加
refactor: 目標判定ロジックを分離
```

使用する type。

- `feat`: 機能追加
- `fix`: バグ修正
- `docs`: ドキュメント
- `style`: 見た目や整形のみ
- `refactor`: 挙動を変えない整理
- `test`: テスト追加・修正
- `chore`: 設定・雑務

### 10.3 コミット前コマンド

プロジェクトに存在する場合、以下を実行する。

```bash
npm run lint
npm run typecheck
npm test
```

存在しない場合は `package.json` を確認し、利用可能な検証コマンドを実行する。

## 11. Codex / Claude Code / AIエージェント作業ルール

AIエージェントは以下を守る。

### 11.1 作業開始時

- まず既存構成を確認する。
- `package.json`、README、設定ファイルを確認する。
- 既存の命名・設計に合わせる。
- 勝手に技術スタックを変更しない。
- 大規模リファクタリングを勝手に始めない。

### 11.2 実装時

- 変更範囲を最小にする。
- 既存ファイルの意味を壊さない。
- 同じ処理を重複させない。
- 型安全を優先する。
- エラー処理を省略しない。
- TODOだけ残して実装完了にしない。

### 11.3 禁止事項

- APIキーや秘密情報の生成・直書き
- 既存コードの無断大量削除
- 理由のない依存ライブラリ追加
- UIの大幅変更を無断で行うこと
- テストや起動確認をせずに完了扱いすること
- 関係ないファイルの整形だけを大量に行うこと

### 11.4 作業完了時の報告

AIエージェントは完了時に以下を報告する。

- 変更したファイル
- 変更内容
- 実行した確認コマンド
- 成功・失敗結果
- 未確認事項
- 次にやるべきこと

## 12. 仕様判断ルール

迷った場合は以下で判断する。

### 12.1 機能追加するか迷った時

追加してよい条件。

- ユーザーの継続率が上がる
- 操作が簡単になる
- 目標達成に直結する
- 既存UIを複雑にしない

追加しない条件。

- 面白そうだが使い道が曖昧
- 初心者にとって操作が増える
- 実装後の保守コストが大きい
- 現時点で検証できない

### 12.2 UIで迷った時

- 説明がなくても使える方を選ぶ。
- 文字より視覚で分かる方を選ぶ。
- 1画面に詰め込むより、段階的に見せる方を選ぶ。
- 初心者が迷わない方を選ぶ。

### 12.3 データ設計で迷った時

- 後から項目追加しやすい構造を選ぶ。
- 既存データを壊さない構造を選ぶ。
- 表示用データと保存用データを混同しない。
- 計算で出せる値を無駄に保存しない。

## 13. Definition of Done

作業完了条件は以下。

- 要件を満たしている
- アプリが起動する
- 対象機能が手動確認済み
- 主要なエラーケースを確認済み
- TypeScriptエラーがない
- Lint/Formatが通る、または未整備の場合は理由を記録している
- 不要なログ・コメントアウトがない
- 変更内容がGitで確認できる
- コミットメッセージが分かりやすい

## 14. 今後追加する可能性が高い機能の設計方針

### 14.1 目標管理

- 目標は長期・中期・短期で分けられる設計にする。
- 目標には達成条件を持たせる。
- 目標に対する行動ログを紐づける。

### 14.2 毎日のチェックイン

- 入力項目は少なくする。
- 毎日続けられる軽さを優先する。
- 入力後すぐにフィードバックを返す。

### 14.3 天使・悪魔判定

判定ロジックは必ず関数として分離する。

例。

```txt
入力: 目標、今日の行動、直近の継続状況
処理: 目標に近づいているかをスコア化
出力: angel / devil / neutral
```

- 判定理由も表示できるようにする。
- 将来的にAI判定に置き換える場合も、ルールベース判定をフォールバックとして残す。

### 14.4 通知

- 通知はしつこくしない。
- ユーザーが時間帯・頻度を選べるようにする。
- ネガティブな通知文を避ける。

### 14.5 ゲーミフィケーション

- 継続日数、バッジ、キャラクター成長などは追加候補。
- ただし、記録すること自体が目的にならないようにする。
- 目標達成行動に結びつく設計にする。

## 18. 最重要メッセージ

このアプリは、機能を増やすだけでは成功しない。  
ユーザーが毎日開き、少し前向きになり、次の行動を1つ起こせることが最重要である。

実装では常に以下を問い続ける。

```txt
この変更で、ユーザーは未来に近づきやすくなるか？
この変更で、操作は簡単になるか？
この変更で、長く使いたくなるか？
```
